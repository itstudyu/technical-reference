What is an API Gateway?

An API Gateway is a server that acts as an API front-end, receiving API requests, enforcing throttling and security policies, passing requests to the back-end service, and then passing the response back to the requester. It serves as a single entry point for multiple microservices.

Kong Gateway Configuration

# Add a service
curl -i -X POST http://localhost:8001/services/ \
  --data "name=user-service" \
  --data "url=http://user-service:3000"

# Add a route
curl -i -X POST http://localhost:8001/services/user-service/routes \
  --data "hosts[]=api.example.com" \
  --data "paths[]=/users"

# Add rate limiting plugin
curl -i -X POST http://localhost:8001/services/user-service/plugins \
  --data "name=rate-limiting" \
  --data "config.minute=100" \
  --data "config.hour=1000"

# Add authentication plugin
curl -i -X POST http://localhost:8001/services/user-service/plugins \
  --data "name=jwt"

NGINX API Gateway Setup

upstream user_service {
    server user-service:3000;
    server user-service-2:3000;
}

upstream order_service {
    server order-service:3001;
    server order-service-2:3001;
}

server {
    listen 80;
    server_name api.example.com;

    # Rate limiting
    limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
    
    # Authentication
    location /auth {
        internal;
        proxy_pass http://auth-service/validate;
        proxy_set_header X-Original-URI $request_uri;
    }

    # User service route
    location /api/users {
        auth_request /auth;
        limit_req zone=api burst=20 nodelay;
        
        proxy_pass http://user_service;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # Order service route
    location /api/orders {
        auth_request /auth;
        limit_req zone=api burst=20 nodelay;
        
        proxy_pass http://order_service;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

Best Practices

üèóÔ∏è Design Principles

  • Single Responsibility: Focus on API management, not business logic
  • Stateless Design: Keep gateway stateless for scalability
  • Fail Fast: Quick error responses to prevent resource waste
  • Backward Compatibility: Support API versioning

üîí Security

  • Defense in Depth: Multiple security layers
  • Principle of Least Privilege: Minimal required permissions
  • Regular Security Audits: Continuous vulnerability assessment
  • Secure Defaults: Security-first configuration

‚ö° Performance

  • Caching Strategy: Implement intelligent caching
  • Connection Pooling: Reuse connections to backend services
  • Asynchronous Processing: Non-blocking request handling
  • Resource Limits: Set appropriate timeouts and limits

üìä Monitoring

  • Comprehensive Logging: Log all requests and responses
  • Health Checks: Monitor gateway and service health
  • Alerting: Set up proactive alerts
  • Distributed Tracing: Track requests across services

‚ö†Ô∏è Common Anti-Patterns to Avoid

Gateway as a Monolith

Putting too much business logic in the gateway makes it a bottleneck

Direct Database Access

Gateway should not directly access databases, only route to services

Chatty API Design

Avoid creating APIs that require multiple round trips

Ignoring Error Handling

Proper error handling and circuit breakers are essential
